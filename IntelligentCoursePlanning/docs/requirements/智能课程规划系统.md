# 课程项目任务书

## 智能课程规划系统

## 项目背景

高校课程规划通常需要同时考虑培养方案、课程先修关系、课程类别学分、开课学期、教学班时间冲突、每学期学分负载以及学生个人偏好。请围绕“智能选课规划”设计并实现一个数据结构项目：系统能够组织课程数据，并在给定专业目标和约束后生成一份可检查、可解释的八学期课程安排或选课建议。

## 数据说明

项目提供一组课程规划数据。当前数据位于 `data/` 目录，主要包括：

- `course_info.csv`：课程信息表，包含 3000 条教学班记录，覆盖 1000 门基础课程。
- `course_time.csv`：开课时间表，每个教学班至少有 1 条上课时间记录。
- `sample_constraints.json`：计算机科学与技术专业的样例规划约束。
- `major_profiles/`：多个专业的样例培养要求，用于测试不同专业和需求下的规划。
- `course_visualization.html`、`interactive_major_plans.html`：课程可视化和参考课表示例。

课程池覆盖计算机、软件、人工智能、数据科学、数学、统计，以及教育学、中文、历史、心理、学前教育、小学教育、英语、艺术等师范类和文科方向。数据中包含课程、教学班、开课时间、教师、教室、容量、先修关系等信息，可用于完成选课规划、冲突检测、课程筛选和可视化展示。

### `course_info.csv`

| 字段 | 字段说明 |
| --- | --- |
| `course_basic_ID` | 课程基础序号。同一门课程的不同教学班共享该字段，例如 `COMS0031121009`。 |
| `course_sp_ID` | 教学班序号。与 `course_basic_ID` 共同组成教学班唯一键，例如 `01`、`02`。 |
| `course_name` | 课程名称。 |
| `department` | 开课院系。 |
| `semester` | 开课季节，取值为 `秋季学期` 或 `春季学期`。 |
| `recommended_term` | 建议/最早修读学期，取值 1 到 8。它不是固定学期，课程可以后移到同季节的更晚学期。 |
| `category` | 课程类别，包括 `专业必修课`、`学科基础课`、`专业选修课`。 |
| `credit` | 学分。 |
| `beg_week` | 开始周数。 |
| `last_week` | 持续周数。 |
| `teacher` | 授课教师。 |
| `classroom` | 上课教室。 |
| `limits` | 选课人数上限。 |
| `prereq_ID` | 前置课程的 `course_basic_ID`。多门先修课用英文分号 `;` 分隔，空值表示无先修课。 |

### `course_time.csv`

| 字段 | 字段说明 |
| --- | --- |
| `course_basic_ID` | 课程基础序号。 |
| `course_sp_ID` | 教学班序号。 |
| `day` | 上课日期缩写：`Mon`、`Tue`、`Wed`、`Thu`、`Fri`、`Sat`、`Sun`。 |
| `beg` | 开始节数。例如上午第三节是 3，下午第一节是 6。 |
| `last` | 持续节数。例如连续 2 节课则为 2。 |

## 输入与约束

系统至少应支持以下输入或配置。输入形式可以是 JSON、表单、命令行参数、代码配置或图形界面控件，但必须能说明清楚。

- 目标专业或目标课程集合，例如 `target_department`、`target_course_scope.required_course_basic_IDs`。
- 每学期学分上限，例如 1-8 学期分别设置最大可选学分。
- 八学期总学分下限，例如 `min_total_credit`。
- 选修学分要求或选修课候选集合。
- 学生偏好或限制，例如不希望周五下午上课、尽量避免晚课、不排周末、大三交换、大四实习等。
- 已修课程、禁选课程或指定必须选择的课程。如果暂不实现，应在报告中说明。

规划时应先确定目标专业或目标课程集合，再完成该集合中的专业必修课和学科基础课，并从选修课中补足总学分要求。

## 项目要求

### 基本选课（40 分）

将目标课程排布在八个学期之内，生成可检查、可解释的课程规划。至少满足：

- 时间冲突处理（10 分）：同一学期内选定课程之间不得存在时间冲突。冲突判断应同时考虑周次、星期和节次；一门课程如果有多个上课时间段，所有时间段都要检查。
- 先修关系处理（10 分）：后继课程必须安排在前置课程之后。建议使用图、邻接表、入度表、拓扑排序或等价机制建模，并能识别缺失前置课、依赖环或无法满足的先修链。
- 每学期学分上限（10 分）：每学期课程学分不能超过输入的学分上限。若上限过低导致无解，应给出原因，而不是输出违反约束的课表。
- 类别与总学分要求（10 分）：目标专业或目标课程集合中的专业必修课、学科基础课应全部完成，并用专业选修课等课程补足总学分下限。输出中应包含总学分、必修/基础学分、选修学分等统计。

若同一门课程有多个教学班，系统应选择其中一个可行教学班；如果所有教学班都冲突，应能说明该课程无法安排或需要调整约束。

### 图形化界面（25 分）

- 支持全部课程信息的导入、展示、可视化查询，以及所选课程信息或规划结果的导出。导入导出格式不做限制，但要在报告中说明。（15 分）
- 支持针对课程字段的筛选功能。例如按学分、持续周数、课程类别、开课学期、院系、教师、教室等字段筛选。（10 分）

图形界面可以是 Web 页面、桌面程序、本地 HTML 可视化工具或其他可交互形式。命令行输出可以作为辅助，但不能完全替代图形化界面要求。

### 实验报告（15 分）

报告应对以下内容进行详尽说明，并提供必要的代码片段、截图、视频或运行结果：

- 项目目录结构、主要模块和运行方式。
- 课程数据、开课时间、教学班、先修关系和学生约束的数据结构设计。
- 规划算法思路，例如拓扑排序、贪心、搜索、启发式排序、多目标评分等。
- 时间冲突、先修关系、学分上限、类别学分和总学分的检查流程。
- 正常样例、边界样例和不可行样例的测试结果。
- 算法复杂度、适用范围、局限性和后续改进方向。

### 拓展内容（20 分）

以下方向可任选。每个方向最高 10 分，累计最高 20 分。拓展功能不能替代基本选课、图形化界面和实验报告的缺失。

#### 前端功能完善（10 分）

在图形化界面实现手动选择与取消选课、保存方案、清空方案、查看冲突原因、重新规划等功能。手动操作后应能重新检查冲突、先修关系和学分约束。

#### 个性化时间安排（10 分）

选课计划支持学生希望在某个时间段上课，或者不希望在某个时间段上课。偏好可以作为硬约束或软约束，但系统应说明哪些偏好被满足、哪些只能部分满足，以及原因。

示例：

- 尽量不要安排周五下午课程。
- 不排周末课程。
- 尽量避免第 11-13 节晚课。
- 不要早 8 和下午第一节。
- 大三交换，第 5、6 学期不安排本校课程。
- 大四实习，第 7、8 学期不安排本校课程，尽量前三年修完毕业学分。

#### AI 选课建议（10 分）

系统可以接入大模型接口，实现自然语言交互。AI 功能应结合真实课程数据和规划结果，而不是只做通用聊天。要求撰写一个格式化的 SKILL 或等价调用说明，用来描述如何把用户自然语言需求转成结构化约束、如何调用规划器、如何解释结果。

```text
User: "我希望前三学期课程压力小一些，每学期不超过18学分，尽量不要安排周五下午的课程，可以怎么选课？"
AI: "我会把每学期学分上限设置为18，并把周五下午设为软避开时段，然后优先安排先修链较长的基础课程..."

User: "我想尽快修完机器学习相关课程。"
AI: "建议先完成数学分析、线性代数、程序设计、数据结构等先修课程，再在满足学分上限的前提下提前安排机器学习和数据方向选修课..."

User: "请对我当前的选课情况给出评价。"
AI: "当前方案总学分达到毕业要求，但第3学期负载偏高，且有2个晚课时段。如果你更重视时间均衡，可以把一门专业选修课后移..."
```

不得在提交文件中暴露真实 API Key。若使用模拟接口或离线示例，应明确说明。

#### 自定义功能实现（10 分）

可以实现其他与智能课程规划相关的功能，例如多方案对比、培养方案完成度分析、课程依赖图、不可行原因诊断、推荐解释、导出日历、教师或教室资源检查等。报告中应阐明该功能的合理性、实现难度、实现逻辑和涉及的数据结构或算法。

## AI 使用说明

可以使用 AI 工具辅助查资料、生成样例、改进代码或整理报告。我们鼓励大家使用现代编码辅助工具，包括但不限于 `Codex`、`Claude Code`、`Qodercli`、`OpenCode`、`Kimi Code`。但最终提交内容必须由小组理解、验证并负责。答辩时应能解释核心数据结构、算法流程和关键代码；如果无法解释，相关功能不应视为完整完成。
