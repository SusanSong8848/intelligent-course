## 一、项目背景

高校课程规划普遍面临"信息量大、约束多、决策难"的问题：
- 1000+ 课程如何按先修关系排序？
- 必修/选修学分如何分配到 8 个学期？
- 开课时间冲突怎么办？
- 交换/实习学期如何隔离？
- 个人时间偏好如何体现？

本系统通过 **C++17 调度引擎 + Web 前端可视化 + AI 选课顾问**，全自动生成 8 学期选课方案。

## 五、算法设计核心思路

### 5.1 整体流程（分阶段说明）

整个项目从零开始，经过以下完整的开发和生产流程：

```
数据加载 → 先修关系图构建 → 必修课拓扑排序 → 贪心学期分配 → 选修学分补足 → 验证 → 导出
```

**第一阶段：数据加载与数据结构设计**

- 从 `course_info.csv`（3000 条教学班记录）和 `course_time.csv`（4563 条时间记录）解析原始数据
- 设计 `TimeSlot`（上课时间段，含 day/beg/last）、`CourseOffering`（教学班）、`CourseBasic`（课程基础，聚合所有教学班和先修关系）、`Constraint`（约束条件，含学分上限/避让时段/偏好时段/封锁学期）四层数据结构
- 使用 `std::map<std::string, CourseBasic>` 构建 `course_map`，以课程 ID 为键，支持 O(log n) 快速查找
- 从 JSON 约束文件加载目标专业配置（必修课集合、选修候选池、每学期学分上限）

**第二阶段：先修关系图构建**

- 遍历 `course_map`，为每门课的先修关系建立有向边
- 构建双邻接表：`adj_list`（先修课 → 后继课程）和 `reverse_adj_list`（课程 → 先修课列表）
- 使用 Kahn 算法进行拓扑排序，同时检测环（如果有环，记录参与环的课程名称）
- 统计图基本信息：1000 节点、664 条有向边、0 环
- 使用 DFS + 记忆化计算每门必修课的"先修链深度"，用于后续调度优先级排序

**第三阶段：课程调度算法（核心）**

- Phase 2：按学期 1→8 迭代，每个学期内筛选出"先修已满足 + 季节匹配 + 推荐学期≤当前"的必修课候选
- 对候选课程按先修链深度降序排列（深度越大越优先，因为它是更多课的瓶颈），同深度再按学分降序
- 对每门候选课，遍历所有教学班，选出季节匹配且时间不冲突的"可行教学班"列表
- 对可行教学班按软约束评分（偏好时段 +1 分，避让时段 -1 分），选得分最高的加入该学期
- 如果学分超限，课程推迟到后续同季节学期（秋季课可后移到 3/5/7 学期，春季课可后移到 4/6/8 学期）
- Phase 3：选修课阶段，按"当前学分从低到高"的学期顺序填补选修课，直到选修学分和总学分均达标

**第四阶段：验证与导出**

- Phase 4：四维度验证——① 时间冲突检查（三维：周次∩星期∩节次）；② 先修关系检查（逐学期验证）；③ 学分上限检查；④ 总学分和类别学分达标检查
- Phase 4.5：分析个性化时间偏好满足度（统计每门课的偏好时段匹配数和避让时段成功避开数）
- Phase 5：生成可解释的规划摘要（含学分统计、每学期负载、未安排课程和冲突详情）
- 最后通过 `json_exporter` 将规划结果序列化为 JSON，供前端读取展示

**第五阶段：前端可视化与交互**

- 前端通过 `fetch` 加载 `schedule_result.json`，渲染 6 个 Tab 页面
- 学分柱状图（ECharts bar chart）+ 8 学期课表卡片 + 时间热力图（8 个独立热力图 + 总览热力图）+ 先修关系力导向图
- 支持手动选课、删除课程、保存/加载方案（localStorage）、查看冲突详情
- AI 选课顾问支持 DeepSeek API 在线模式和离线关键词模拟模式

**完整数据流**：
```
CSV 文件 → C++ 解析 → CourseDataset → Scheduler 调度 → ScheduleResult
                                                          ↓
前端可视化 ← fetch JSON ← json_exporter 导出 ← 验证通过
```

### 5.2 核心数据结构

- **邻接表双图**：`adj_list`（pre → successors）+ `reverse_adj_list`（course → prerequisites）
- **软约束评分**：`score = 偏好时段匹配数 - 避让时段命中数`，选得分最高的教学班
- **负载均衡贪心**：选修课按"当前学分从低到高"的学期顺序分配

### 5.3 时间复杂度

O(N + E + T×C×O) → 1000 节点 664 边，8 学期，实测 < 0.5 秒

## 七、答辩准备问答

### 技术类

**Q1：如何保证先修关系不违反？**
A：Kahn 拓扑排序 + 按学期逐层分配，`are_prerequisites_satisfied()` 在每门课排入前检查所有先修课是否已安排在更早学期。

**Q2：如果所有教学班都冲突怎么处理？**
A：返回 nullptr，留给后续学期尝试。Phase 2.5 统一收集未安排课程并输出原因。

**Q3：选修课为什么能均衡分配到 8 个学期？**
A：`fill_electives` 按学期当前学分排序（`std::sort`, ascending），选修课优先填入负载最低的合法学期。

**Q4：内嵌 JSON 解析器为什么不用 nlohmann/json？**
A：零外部依赖策略，使用 C++17 `std::variant` + 递归下降解析，避免 FetchContent 网络下载失败。

**Q5：浮点数学分会不会超限？**
A：所有学分上限判断使用 `max_credit + 0.001` epsilon 容差，防止 0.1+0.2 ≠ 0.3 的精度问题。

### 功能类

**Q6：AI 选课顾问怎么工作的？**
A：用户输入自然语言 → 关键词意图识别（5 类意图 × 正则匹配）→ 构造 system prompt（含当前课表的真实数据）→ DeepSeek API → 返回自然语言建议。详见 `docs/AI_SKILL.md`。

**Q7：前端怎么解决 file:// 协议下 fetch 报错？**
A：Python 简易 HTTP 服务器 + `start.bat` 双击启动，自动打开浏览器到 `http://localhost:8000`。

**Q8：热力图的三色是怎么实现的？**
A：ECharts visualMap 只支持单色渐变。三色通过给每个 data point 单独设置 `itemStyle.color`，必修红 `#a24f4a`/基础黄 `#b07a20`/选修绿 `#4d7c3f`。

### 项目管理类

**Q9：AI 在项目中参与了哪些工作？**
A：详见实验报告"AI 协作声明"章节。AI 辅助算法框架设计、前端图表配置、文档初稿生成。所有核心逻辑（双邻接表设计、软约束评分、负载均衡贪心）由本人 review 并调整，测试全部通过。

**Q10：项目最大的技术难点是什么？**
A：① 先修链深度计算（DFS + 记忆化），避免重复遍历；② 软约束评分与硬约束的协同（偏好 vs 冲突检测）；③ 前端 ECharts 与数据绑定的交互逻辑（热力图三色渲染、依赖图 hover 详情）。
